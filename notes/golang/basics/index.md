# Введение в Go

## 1. История и философия

*   Разработан в **Google**.
*   Выпущен в Open Source в **2009 году**.
*   Является одним из ведущих языков программирования и **языком №1** в экосистеме **CNCF** (Cloud Native Computing Foundation).

**Проблемы, которые решал Go при создании:**
*   Низкая скорость разработки и медленная сборка проектов.
*   Отсутствие контроля над зависимостями.
*   Сложность понимания кода.
*   Сложность разработки инструментов (тулинга).
*   Сложная реализация многопоточности.
*   Высокая стоимость обновлений.

**Ключевые решения:**

1.  **Быстрая компиляция:**
    *   Каждое приложение — это модуль, состоящий из пакетов.
    *   Пакет — минимальная единица компиляции.
    *   Go компилирует все пакеты параллельно, и каждый пакет обрабатывается только один раз.

2.  **Контроль над зависимостями:**
    *   **Неиспользуемый импорт — это ошибка компиляции.**
    *   Граф зависимостей не имеет циклов и лишних веток, что упрощает логику.

3.  **Читаемость языка:**
    *   **Форматирование кода (`gofmt`) является частью синтаксиса.**
    *   Благодаря этому весь Go-код выглядит одинаково, что снижает стоимость интеграции в новую кодовую базу.

## 2. Go Runtime

*   **Runtime** — это набор кода и инструкций, необходимых для работы приложения и обеспечения возможностей языка.
*   В отличие от Rust (где рантайм почти отсутствует) или C/C++ (где он представлен стандартной библиотекой), в Go, Python, C# **runtime является неотъемлемой частью** любой программы и исполняется в фоне.
*   **Задачи Go Runtime:**
    *   Инициализация памяти.
    *   Работа планировщика (шедулера) для горутин.
    *   Работа сборщика мусора (GC).
    *   Защита разработчика от некоторых ошибок (например, доступ по nil-указателю).
    *   Предоставление контроля над исполнением программы.

*   **Go — язык с автоматическим управлением памятью:**
    *   Используется Garbage Collector (GC), построенный на модели **mark-sweep** (пометка и очистка).
    *   GC в Go является высокопроизводительным и хорошо отлаженным.

### Как работает сборщик мусора (GC) в Go

#### Основной принцип

Go использует **конкурентный трассирующий сборщик мусора** с короткими паузами, известными как "stop-the-world" (STW). Основной алгоритм — **Concurrent Mark and Sweep (Параллельная пометка и очистка)**.

Процесс делится на два ключевых этапа:
1.  **Mark (Пометка)** — поиск и пометка всех "живых" (используемых) объектов в памяти.
2.  **Sweep (Очистка)** — удаление всех "мёртвых" (недостижимых) объектов и освобождение занимаемой ими памяти.

#### Фазы работы GC

1.  **STW-пауза: Начало пометки (Mark Setup)**
    *   Все активные горутины ненадолго приостанавливаются.
    *   Включается специальный механизм — **барьер записи (write barrier)**, который отслеживает изменения указателей во время конкурентной фазы.
    *   Эта пауза очень короткая (обычно доли миллисекунды).

2.  **Конкурентная пометка (Concurrent Mark)**
    *   GC сканирует память и ищет все живые объекты, начиная с корневых (глобальные переменные, стеки горутин).
    *   Эта фаза выполняется **параллельно с пользовательским кодом**, что составляет основную часть работы GC.
    *   Благодаря барьеру записи, GC корректно обрабатывает новые объекты, созданные во время этой фазы.

3.  **STW-пауза: Завершение пометки (Mark Termination)**
    *   Снова короткая пауза для завершения всех оставшихся задач пометки.
    *   Барьер записи отключается.

4.  **Конкурентная очистка (Concurrent Sweep)**
    *   GC проходит по всей куче и освобождает память, занятую непомеченными (мёртвыми) объектами.
    *   Эта фаза также выполняется **параллельно** и не блокирует выполнение основного кода.

#### Настройка с помощью `GOGC`

| Значение `GOGC` | Эффект                                                                          |
| :-------------- | :------------------------------------------------------------------------------- |
| `100` (дефолт)  | Новый цикл GC начнется, когда размер кучи увеличится на 100% после предыдущего GC. |
| `> 100`         | GC запускается реже → **больше используется памяти**, но меньше накладных расходов.  |
| `< 100`         | GC запускается чаще → **меньше используется памяти**, но больше накладных расходов.   |
| `off`           | Полностью отключает GC (используется крайне редко, в основном для отладки).       |

# Основы языка Go

## Типы данных

*   **Элементарные типы и их нулевые значения (Zero Values):**
    *   `bool`: `false`
    *   `string`: `""` (пустая строка)
    *   `int`, `uint` (разных размеров): `0`
    *   `float`: `0.0`
    *   `complex128`: `0 + 0i`
    *   `byte` (алиас для `uint8`): `0`
    *   `rune` (алиас для `int32`, представляет Unicode point): `0`

*   **Комплексные типы:**
    *   `Pointer`
    *   `Struct`
    *   `Array`
    *   `Slice`
    *   `Map`
    *   `Function`
    *   `Interface`
    *   `Chan`

## Ключевые концепции

*   **Горутины (Goroutines):**
    *   Легковесный поток, которым управляет Go Runtime. Любое Go-приложение состоит как минимум из одной (главной) горутины.
    *   Запускаются с помощью ключевого слова `go`.

*   **Функции — "first class citizen":**
    *   Функции являются типами и значениями, их можно передавать как аргументы, возвращать из других функций и присваивать переменным.

*   **Собственные типы:**
    *   Можно создать новый тип на основе любого существующего: `type Category string`.
    *   Это позволяет добавлять методы к простым типам и создавать перечисления (Enum) с помощью констант.
    *   Собственный тип не является взаимозаменяемым с базовым типом без явного приведения.
    *   **Алиасы (`type Country = string`)** являются полностью взаимозаменяемыми, но не могут иметь собственных методов.

*   **Структуры и методы:**
    *   **Структура** — это объединение именованных полей.
    *   Поля, начинающиеся с заглавной буквы, являются **публичными** (экспортируемыми). С маленькой — **приватными**.
    *   **Встраивание (embedding):** Позволяет встроить один тип в другой, наследуя его поля и методы.
    *   **Методы** — это функции, связанные с определенным типом.
    *   **Ресивер (получатель) метода:**
        *   `func (p Product) Inc() {}` — ресивер по значению. Метод работает с копией объекта.
        *   `func (p *Product) SetCountry() {}` — ресивер по указателю. Метод работает с оригинальным объектом и может его изменять. **Модификация объекта возможна только в методах с ресивером-указателем.**

*   **Интерфейсы (Interface):**
    *   Определяют набор методов.
    *   Тип удовлетворяет интерфейсу **неявно**, если он реализует все его методы.
    *   Соответствие проверяется во время компиляции.
    *   Вся стандартная библиотека и экосистема Go построены на интерфейсах (`io.Reader`, `error`, `context.Context` и др.).

*   **Дженерики (Generics):**
    *   Позволяют писать функции и типы, которые работают с набором типов.
    *   Набор допустимых типов определяется с помощью интерфейса-ограничения (`interface { ~int | string }`).

*   **Ошибки (Errors):**
    *   В Go ошибка — это просто значение, которое реализует интерфейс `error` с одним методом `Error() string`.
    *   **Возврат нескольких значений** (`value, err`) — стандартный способ обработки ошибок.

*   **Контекст (Context):**
    *   Тип `context.Context` — стандартный инструмент для контроля исполнения длительных задач, передачи сигналов отмены и мета-информации (например, trace ID).
    *   Популярный паттерн — передавать контекст первым параметром в функцию.

*   **Defer, Panic, Recover:**
    *   **`defer`:** Гарантирует выполнение функции при выходе из текущей функции (независимо от того, как она завершилась). Используется для закрытия ресурсов (файлов, соединений).
    *   **`panic`:** Вызывает панику, останавливая обычное выполнение горутины. `defer` функции при этом выполняются. Используется для действительно исключительных ситуаций.
    *   **`recover`:** Позволяет "поймать" панику и возобновить нормальное выполнение. Имеет смысл только внутри `defer` функции.

*   **Cross-компиляция и CGO:**
    *   Go поддерживает кросс-компиляцию для разных ОС и архитектур "из коробки".
    *   **CGO** позволяет вызывать функции из C-кода, но это усложняет сборку и считается "не идиоматичным" Go.

# Массивы, слайсы и карты

## 1. Array vs Slice (Массивы и Слайсы)

### Array (Массив)

*   **Определение:** Структура данных с фиксированной длиной, которая хранит множество элементов одного типа.
*   **Инициализация:**
    *   С нулевыми значениями: `var arr1 [4]int`
    *   С указанием значений: `arr2 := [3]int{1, 2, 3}`
    *   С автоматическим определением размера: `arr3 := [...]int{4, 5, 6, 7}`
*   **Особенность:** Размер и тип массива являются частью его сигнатуры. Это означает, что массивы разного размера считаются разными типами.
*   **Передача в функции:** Массив передаётся по значению, а не по ссылке. Это означает, что при передаче массива в функцию создается его полная копия. Это может быть неэффективно для больших массивов.
*   **Когда использовать массивы?**
    *   Для оптимизации работы с памятью.
    *   Для сравнения через оператор равенства (`==`).
    *   В качестве ключа для map.
    *   Для паддинга (выравнивания) памяти в структурах.
    *   Массивы являются основой для слайсов.

### Slice (Слайс)

*   **Определение:** Динамический массив, который предоставляет более гибкий интерфейс для работы с последовательностями данных.
*   **Характеристики:**
    *   Типизируется только по типу данных, но не по размеру.
    *   Каждый слайс имеет под собой массив, который скрыт от пользователя.
*   **Структура слайса:** Слайс представляет собой структуру, содержащую:
    *   `array unsafe.Pointer`: Указатель на начало нижележащего массива.
    *   `len int`: Длина (количество элементов в слайсе).
    *   `cap int`: Емкость (количество элементов в нижележащем массиве от указателя до конца).
*   **Аллокация (создание):**
    *   Используется функция `make([]T, len, cap)`.
    *   `len` - количество заполненных элементов.
    *   `cap` - опциональный аргумент, общее количество аллоцированных элементов (`cap >= len`).
*   **Добавление элементов:**
    *   Функция `append` принимает слайс и один или несколько элементов, возвращая новый слайс.
    *   Если при добавлении элементов превышается `cap`, происходит реалокация - создается новый, больший массив, и данные копируются в него.
*   **Slicing (создание подслайса):**
    *   Слайс можно создать на основе другого слайса или массива: `sub := ids[1:2]`.
    *   **Важно:** Изменения в подслайсе влияют на оригинальный слайс, так как они ссылаются на один и тот же нижележащий массив.
    *   **Потенциальная проблема:** Если функция возвращает небольшой срез большого файла (например, `os.ReadFile`), сборщик мусора не сможет освободить память, занимаемую всем файлом, пока существует ссылка на этот срез.
    *   **Решение:** Явно скопировать интересующие данные в новый слайс: `c := make([]byte, len(b)); copy(c, b)`.
*   **Преобразование:**
    *   **Слайс в массив:** Копирование элементов в цикле.
    *   **Массив в слайс:** `slice := array[:]` (создает слайс, указывающий на тот же массив) или `sliceCopy := make([]int, len(array)); copy(sliceCopy, array[:])` (полное копирование).

## 2. Maps (Карты)

*   **Определение:** Структура данных, хранящая пары "ключ-значение".
*   **Характеристики:**
    *   Реализована на основе хеш-таблицы.
    *   Порядок элементов не гарантирован.
    *   Нельзя взять ссылку на элемент карты.
    *   **Не является concurrent-safe** (небезопасна для использования в нескольких горутинах без синхронизации).
*   **Инициализация:**
    *   `var ageByName map[string]int` (nil-карта, запись в нее вызовет панику).
    *   `ageByName := make(map[string]int, 10)` (с начальной емкостью).
    *   `ageByName := map[string]int{"Mike": 1, "Aaron": 2}` (литерал карты).
*   **Получение значений:**
    *   При чтении может возвращать два значения: само значение и `bool`-флаг, указывающий на наличие ключа: `age, ok := ageByName["Mike"]`.
*   **Тип ключа:**
    *   **Допустимые типы:** Любые типы, для которых определен оператор сравнения (`==`), например: `int`, `string`, `float`, указатели, массивы, структуры (если все их поля сравнимы), интерфейсы, каналы.
    *   **Недопустимые типы:** Слайсы, функции, карты и структуры, содержащие эти типы.
*   **Удаление элементов:**
    *   `delete(ageByName, "Aaron")` - удаляет значение по ключу.
    *   `clear(ageByName)` - удаляет все значения из карты.
*   **Внутренняя реализация:** Начиная с Go 1.24+, используется реализация на основе SwissTable. Не стоит полагаться на конкретную внутреннюю реализацию.

### Часть 1: Основные понятия (Что такое Map и Хэш-таблица?)

**1. Определение и Цель `map`**
*   **Абстрактная структура данных:** `map` (также известный как ассоциативный массив, словарь или хэш-таблица) — это структура данных для хранения пар "ключ-значение".
*   **Основные операции:** Она должна эффективно выполнять три ключевые операции:
    *   `Insert(ключ, значение)`: Вставка или обновление значения по ключу.
    *   `Lookup(ключ)`: Поиск значения по ключу.
    *   `Remove(ключ)`: Удаление пары по ключу.
*   **Ключевое требование:** Все эти операции должны выполняться за **константное время (O(1))** в среднем. Это означает, что скорость операции не должна зависеть от общего количества элементов в `map`.

**2. Проблема простого подхода**
*   Наивная реализация `map` могла бы использовать обычный массив или срез, где каждый элемент — это пара ключ-значение.
*   **Недостаток:** Поиск (`Lookup`) в такой структуре потребовал бы последовательного перебора всех элементов до нахождения нужного ключа. Это приводит к **линейной сложности (O(n))**, что нарушает требование о константном времени и делает структуру очень медленной при большом количестве данных.

**3. Решение: Хэш-таблицы и Хэш-функции**
*   В основе эффективной реализации `map` в Go (и многих других языках) лежит **хэш-таблица**.
*   **Основная идея:** Преобразовать ключ любого типа в целочисленный индекс с помощью специальной **хэш-функции**.
    *   `индекс = hash(key) % размер_таблицы`
*   **Результат:** Все основные операции (`Insert`, `Lookup`, `Remove`) выполняются примерно за O(1) времени.
*   **Важно:** Хэш-функция должна быть:
    *   Детерминированной: один и тот же ключ всегда даёт один и тот же хэш.
    *   Быстрой.
    *   Обеспечивать равномерное распределение.
    *   Иметь ограниченный диапазон (соответствовать размеру таблицы).
    *   Не быть криптографически стойкой (требуется производительность, а не защита).

### Внутреннее устройство map в Go

*   Структура `hmap` — заголовок карты:
    *   `count` — количество элементов
    *   `B` — степень числа бакетов (2^B бакетов)
    *   `hash0` — seed хэш-функции
    *   `buckets` — массив указателей на бакеты `bmap`
*   **Бакет `bmap`:**
    *   Содержит до 8 пар ключ-значение.
    *   Если элементов больше, создается overflow bucket.
*   **Рост карты:** происходит инкрементально, копируются только элементы из затронутых бакетов.

### Ответы на главные вопросы

1.  Почему заранее аллоцировать память для слайса или карты?  
    Чтобы не делать многократное расширение внутреннего массива при добавлении новых элементов.

2.  Почему порядок обхода карты в Go случайный?  
    Go использует эвакуацию и рандомизацию при итерации для защиты от хардкод-зависимостей и гонок.

3.  Почему нельзя взять указатель на элемент карты?  
    Это ограничение связано с безопасностью работы сборщика мусора и возможным перемещением элементов в памяти.

# Организация проекта и экосистема

*   `/cmd` — точки входа (main.go)
*   `/internal` — приватный код
*   `/pkg` — библиотечный код
*   `go.mod` — описание модуля и зависимостей
*   Экосистема:
    *   Родной toolchain: `go build/run/test/fmt`
    *   Go Modules
    *   Стандартная библиотека
    *   Open Source библиотеки (`awesome-go`)
